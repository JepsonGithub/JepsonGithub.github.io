从业务层面看小程序

微信希望每天各种的需求, 都能通过微信解决

小程序希望有着服务号一样的功能, 跟应用程序一样的体验

它适用于用完即走的应用, 比如说: 打车, 订餐, 买票, 话费充值等…

小程序不太适合中大型游戏, 使用频率特别高的应用(知乎), 一定要是用完即走, 即开即用的应用才是小程序的主场

开发包必须小于 1 M

小程序有个限制就是应用开发包大小必须小于1M (这也是为什么不适合应用于大型应用的原因之一)
所以说小程序不用安装就能使用, 其实是不准确的, 因为其实还是下载了1M以内的应用包, 只是比较小, 用户感知不到而已

总之一句话, 小程序希望实现让应用触手可及, 即用即开, 用完即走

从技术层面看小程序

小程序不需要下载安装, 可以通过微信打开, 那他是怎么实现的呢?

小程序是要运行在微信内部的, 是微信提供的一个平台, 所有的规则都是微信制定的, 包括所有的标签用法, 标签展示效果, 基本js语法等等, 借助微信这样的运行环境, 就能开发出能在微信中跑的应用!

那么所有的用户只要安装了微信, 所有的应用都可以触手可得

开发者不用过多的考虑多终端的兼容性处理问题 (小程序对 ES6 以及 flex 的支持非常的好, 所以写起代码来会非常爽 )

微信小程序初体验

我们会发现小程序自带的样式就是很好看的不需要我们去定制.
本身web端基础样式会丑是由于历史原因, 从一开始出来的时候, 这些控件就很丑.
小程序就不一样了, 他是一个新的平台, 所有样式什么的不受web端基础样式的限制, 很多控件默认样式就比较好看了, 这个是比较爽的

小程序与web开发有思维上的共通, 从技术层面有很多不一样的地方, 还是需要注意的 (比如: 小程序中没有dom, 没有window, 没有ajax等, 它有自己的一套渲染语法, 有自己的request请求语法)

开发工具下载地址: https://mp.weixin.qq.com/debug/wxadoc/dev/devtools/download.html?t=2018125

  • 安装开发工具, 用绑定的微信账号扫一扫

基本配置

小程序中约定大于配置: 小程序中的开发方式以及文件结构目录, 都是约定好了的, 每个目录什么作用等等, 不需要过多的配置, 核心就四个文件类型,

  • js文件
  • json(配置文件)
  • wxml(页面标签等, 类似于html)
  • wxss(页面样式控制,类似于css)

比较简单, 目录结构说明什么的文档都有
文档地址: https://mp.weixin.qq.com/debug/wxadoc/dev/framework/config.html

全局配置 (app.json)

  • 配置顶部导航条文字: "navigationBarTitleText": "哈哈"
  • 配置 bar颜色: "navigationBarTextStyle":"white" 只支持 白色 和 黑色
  • 配置下拉刷新 "enablePullDownRefresh" boolean
  • 配置debug 为 true, 会多很多 log 输出

tips: 注意页面的配置会优先于全局的配置, 也就是页面中json里面配的优先级会更高, 会层叠掉全局的配置

页面配置 (页面.json)

  • 页面中的配置有个小细节: 只能配置窗口相关的配置
  • 页面中最常用的标签: view 相当于div, text 相当于 span

逻辑层js的小 tips

注意: 在小程序中, 没有 window
因为所谓的 bom 和 dom 都是相对有浏览器这个平台而言的, 所以在小程序这个平台中是没有 window的

应用的生命周期

小程序采用 common.js 规范, 每一个js文件都有单独的作用域, 整个应用有自己的生命周期, 在不同的周期会调用不同的回调函数

App 中注册的全局对象中可以添加任意的成员, 这个app对象在全局是共享的, 在其他页面js中, 通过getApp()可以获取全局的 app 应用程序实例对象, 就可以拿到里面的存的属性和方法了

  • 可以通过 onShow 判断 options 中的 scene编号, 来判断当前是通过哪种方式进行小程序的, 用以做不同的处理
  • app对象是全局共享的, 添加的成员, 将来通过 getApp() 方法可以直接获取到 app 应用程序实例对象

页面的生命周期

每个页面的有自己的生命周期, 包括页面的加载, 渲染, 卸载等等…

数据绑定和 wxs

微信小程序中是单项数据绑定的, 数据源中定义的数据, 可以直接通过小胡子语法映射到页面中

tips: 数据源只能暴露数据, 不能暴露函数, 就算在数据源中写了函数, 在页面中也是无法调用的, 这跟我们传统的 vue 中的写法, 就有区别了, 要注意!!!

那么如何在视图层定义一个函数呢 ?
wxs 标签类似于咱们 js 中的 script 标签, 可以定义一些行内脚本, 这里的代码必须遵循 commonjs 规范

控制属性(条件渲染)

wx:if wx:elif wx:else

tips: 上面这种做法虽然可以, 但是这样做不太好,

对于频繁需要切换显示状态的元素, 不应该用 wx:if, 应该用 hidden 性能会更高, 因为 wx:if 会不断创建销毁元素, 而hidden是切换 元素的显示状态

hidden

hidden 根据boolean 值, 切换显示隐藏状态, 相对于上面wx:if这种操作方式, 性能效率更高

block 包装元素

block 只是一个包装元素, 不会页面中已经存在的其他元素结构造成影响

事件处理

官网地址: https://mp.weixin.qq.com/debug/wxadoc/dev/framework/view/wxml/event.html

在组件中绑定一个事件处理函数, 事件前加上 bind 即可

如bindtap，当用户点击该组件的时候会在该页面对应的Page中找到相应的事件处理函数

默认会事件冒泡

不想冒泡, 正常以前的方法(e.stopPropagation, return false)不会起作用了, 需要使用新的关键字 catch

事件前面加上 catch 即可注册不冒泡的事件

比如: 用 catchtap 代替 bindtap 绑定事件, 就不会冒泡了

自定义属性

  • 通过 data-… 来传递值, 必须以 data- 开头
  • 可以通过 data- 属性给事件处理函数, 传递额外的参数

通过事件对象 e.currentTarget.dataset 拿到所有的自定义属性集合
而且如果自定义属性有连字符, 都会被去掉, 变成驼峰命名

单向数据流

在微信小程序中的数据源和页面的映射是单向的, 数据源中的数据改变后会映射到页面中, 但是页面中的数据变化了, 不会直接改变数据源中的数据, 如果希望实现双向数据绑定, 还是需要通过 js 来 setData 来手动改变数据源的

setData 设置多层级数据源的数据

WXSS

weui 有一套专门针对小程序做的 ui 地址: https://weui.io/
正常和 css 一样写就可以了,
单位可以用 px 也可以用 rpx, rpx和rem很像
小程序把页面当成了宽度750的尺寸(按照iphone6的尺寸来的), 你用 rpx 作为单位, 就相当于在宽度 750 的尺寸中布局即可, 写成 375rpx 就相当于设成 宽度 50%了, 且可以自动在不同屏幕下适配, 使用起来非常方便

request 请求

小程序中因为是手机端, 不存在浏览器端才有的跨域问题, 所以直接通过官方提供的 wx.request 方法就可以进行请求获取数据

但是, 小程序自身为了安全有个限制, 你所请求数据的网站地址, 必须配置在小程序管理后台 -> 设置 -> 开发设置 -> 服务器域名中, 且 必须是 https 协议, 域名必须要备案.

  • 当然小程序在开发工具中测试时, 可以不校验 (但是真实上线还是要备案,要https协议才能使用的, 这只不过时掩耳盗铃罢了)
