模块化的标准

为什么要有标准?
让模块拥有更好的通用性!

  1. CommonJS: node.js 中实现模块所遵循的标准

  2. AMD: Async Module Definition 异步模块定义, 代表: require.js, 表示所有的模块化js, 是由一种异步的方式加载进来的
    依赖前置: 在一开始就将所有的依赖全部加载

  3. CMD: Common Module Definition 通用模块定义 sea.js
    依赖延迟: 在需要的时候再去加载依赖项

requirejs 模块定义和引用

  • 1 定义模块
    在require.js 中, 模块也是分成一个个文件保存的, 而且每个模块都有自己单独的作用域
    定义模块的时候, 需要使用 requirejs 提供的方法 -> define方法
    语法: define(“模块名称”, [“模块依赖项”], function() {})

  • 2 引用模块
    在 requirejs 中引用一个模块, 使用 require.js 提供的函数 require 进行引用
    语法: require( [“模块文件的路径(不带.js后缀)”], function() { … } )
    后面的函数是, 模块加载成功之后的回调函数

导出对象

定义模块的时候, 有导出项的时候
如果有导出项, 只需要将导出项 return 即可

导入多个模块, 就是数组内写多个值, c, a, b 按顺序会接收模块的return 导出项, 如果没有导出, 那么就会是 undefined, 不会影响接收的顺序
所以一般我们会把没有返回值的模块放最后, 这样就不需要写全所有的形参来接收每个模块的值了, 因为 undefined 是没必要接收的

模块的依赖项

在定义一个模块的时候, 这个模块可能会使用到其他模块的内容, 这些其他模块就可以称作是当前模块的依赖项

  • need.js: 定义一个有依赖项的js模块

  • 使用 need.js 模块, 会自动加载自己的依赖项

模块路径查找问题

在requirejs 中模块路径的查找方式, 一共分三种情况

1 当前文件路径为参照(相对路径)

  • 最简单的, 不做任何配置, 直接以当前文件的路径为参照

2 配置data-main

  • 如果给加载requirejs的script标签设置了 data-main 属性, 那么main.js内模块的查找, 会以main.js的路径为基础进行查找, 而不是html页面为基础s
    data-main 指定的文件, 会在 requirejs加载完毕后, 通过异步的方式加载, 并执行里面的代码, 这个属性可以称为是入口文件, 刚一开始就要执行的代码都可以放到这个文件中

3 配置 config 基础路径

  • 如果通过 require.config 方法配置了基础路径, 那么所有的模块路径查找都会以这个路径基础为参照, 而且一般都被设置成绝对路径

引用模块的时候, 模块路径 baseUrl + 引用写的路径 才是真正的路径

如果觉得每次都写全路径很麻烦, 可以在 config 中通过 paths 配置路径的别名

// 注意: 第三种方式中,
// 找文件的规则是: baseUrl + 文件路径 或者 baseUrl + paths
// 如果引用模块的时候, 路径写法是如下 3 种形式之一, 就不遵守上面的规则了
// 1. 文件路径 加上了 .js 后缀
// 2. 文件路径 是以 / 开头的绝对路径
// 3. 文件路径 以 http:// 或者 https:// 开头的
